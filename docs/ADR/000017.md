# upstream を複数インスタンス化し、round-robin LB ＋ active health check で分散する

## Status
proposed

## Date
2026-06-21

## Affected Components
- server runtime（`plecto-server`）— 1 upstream を複数アドレスへ拡張。round-robin で分配し、background の **active health check** が各インスタンスを定期 probe して unhealthy を eject・healthy で復帰
- manifest — `[[upstream]]` に複数 `addresses` と `health`（probe `path` / `interval_ms` / healthy・unhealthy 閾値）を宣言

## Context

M2 slice 1（[[000013]]）は 1 route 1 upstream（1 アドレス）で、インスタンス間 LB が無い。可用性・スケールには複数 upstream インスタンスへの分散と、落ちたインスタンスを避ける health 判定が要る。plan §6.10 は「単一ノード・ファースト、分散はオプトイン」。LB は単一ノード内の upstream 分散であり、ノード間分散（foca/openraft）とは別レイヤ。

## Decision

- **upstream を複数アドレスに拡張する。** `[[upstream]] name=... addresses=["h1:p","h2:p", …]`。route は従来どおり upstream を名前で指す。
- **分配は round-robin。** healthy なインスタンス集合を round-robin で選ぶ。
- **health は active health check を採る。** background タスクが各インスタンスを `health.path` へ `interval_ms` ごとに probe し、連続成功/失敗が閾値に達したら healthy/unhealthy を切り替え、unhealthy は分配集合から **eject**、回復で復帰。全インスタンス unhealthy のときは fail-closed（502/503）。
- **状態は単一ノードローカル。** ノード間の health 共有は §6.10 のとおり opt-in の後続。

検討した代替案と却下理由:
- **passive ejection のみ**（実リクエストの接続失敗/5xx で一時 eject、追加 probe なし）— 単純だが「次の実リクエストが踏むまで落ちを検知できない」。**ユーザ判断で active を採用**（落ちを先回り検知）。passive は active の補完として将来併用しうる。
- **least-conn / EWMA 分配** — 偏りに強いが per-instance の接続数/応答時間の状態追跡が要る。round-robin を先行し、偏りが問題化したら導入。却下（後続）。
- **分散（ノード間）health/LB を今入れる** — §6.10 の単一ノード・ファーストに反する。却下（opt-in 後続）。

本決定は plan §6.10（単一ノード・ファースト、分散は opt-in）と整合し、fail-closed（全 unhealthy で upstream に流さない）を保つ。

## Consequences

### Pros
- 複数 upstream インスタンスへ分散でき、可用性・スケールが上がる。
- active health で落ちたインスタンスを**実リクエストが踏む前に**回避できる。
- 単一ノードに閉じ、運用が単純。

### Cons / Tradeoffs
- active probe は upstream ごとに background タスク ＋ 設定（path/interval/閾値）と probe トラフィックを増やす。
- round-robin は接続数/負荷の偏りに弱い（重いリクエストが偏ると不均衡）。
- **再検討条件**：(a) 偏りが問題化したら least-conn/EWMA；(b) passive ejection を active と併用；(c) **upstream TLS**（https upstream への rustls client、[[000014]] と連動）；(d) ノード横断の health/LB は分散レイヤ（foca/openraft）の opt-in。
