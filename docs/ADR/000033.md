# native L7 rate limiting を fast path に導入し、filter に貸す host-ratelimit とは別の粗粒度ベースラインとする

## Status
accepted

## Date
2026-07-01

## Affected Components
- server runtime（`plecto-server`）— `proxy_core_inner` が `normalize_path` + `find_route` の**直後・request-side filter chain の前**で route の native limiter を consult。token が無ければ pick せず **429**（`x-plecto-fault: rate-limited`、body `rate limit exceeded`、`Retry-After`）で即返す。token-bucket の計算は host-native で完結し WASM 境界を跨がない（filter chain とは独立した素のデータプレーン経路）。`peer.ip()`（接続 peer）を key にする
- control plane / manifest（`plecto-control`）— `[[route]]` に任意セクション `[route.rate_limit]`（`rate`（req/sec）+ `burst`（capacity）、`key` = `route` または `client-ip`、既定 = 無制限・key 既定 = `route`）。`build_active` が manifest から limiter を構築（`request_timeout` / circuit breaker と同列、health の外）。`rate == 0` / `burst == 0` は build 時に `ControlError::InvalidRouteRateLimit` で fail-closed 拒否
- extension plane（`plecto-host`）— `apply_bucket`（per-key `(tokens, last_refill_ms)` の token-bucket math、[[000005]]）を pub 化し、native limiter と per-filter `host-ratelimit`（[[000026]]）で**同一実装を共有**。純粋関数なので WASM 境界も filter-state KV も跨がない
- observability — `plecto_rate_limited_total` counter を fast-path metrics に追加（[[000009]]）

## Context

現在 Plecto に存在する rate limit 機構は **filter に貸す `host-ratelimit`（[[000026]]）の 1 つだけ**である。これは host 側の token bucket を WASM フィルタが `try-acquire(key, cost)` で consult する **filter capability** であり、spec（capacity / refill）は per-filter で manifest に置かれ、フィルタが「consult するか・どのキーで・cost いくつで」を判断する。つまり rate limit は**フィルタが効いているルートにしか掛からない**。

ここに穴がある——**フィルタを差し込んでいない素のルートには、リクエストレートに対する built-in な DoS 保護が何も無い**。filterless route は upstream へ無制限にリクエストを通す。すべてのルートに最低限のレート上限を掛けたければ、現状では**そのためだけに WASM フィルタを 1 枚書いて全ルートに配線する**必要があり、native proxy が Tier-0 で備えるべき粗粒度のレート防御が欠けている。

役割駆動の native 採用基準（Fork 6）に照らすと、**リクエストレートの上限は per-request の filter decision では表現できない横断的データプレーン関心事**である。レートは「このルート / この client-IP に向かう全リクエストにまたがる共有カウンタ」であり、global counter は native に置く（[[000029]] の placement rule「global counters are native」、Stage C）。これは [[000028]]（circuit breaker = upstream の in-flight cap）が「飽和の保護」を native に置いたのと同じ系譜の、**inbound 方向のレート保護**である。

本件は DoS 対策ファミリの一部でもある。[[000027]] が host-state quota と inbound リソース上限（接続数・ヘッダサイズ・body 上限など）で**量**を bound したのに対し、本 ADR は**レート（時間あたりのリクエスト数）**を bound する——両者は別軸で相補する。

設計を詰める過程で外部事例を裏取りした: Envoy の **local** rate limit は token bucket（`max_tokens` = burst / `tokens_per_fill` / `fill_interval`）で、超過時は既定 **429** を返す（fixed/sliding window ではない）。Rust の `governor`（GCRA）は keyed limiter を提供するが keyed map を自動で bound せず、かつ新規 dependency subtree（`dashmap` 系）を持ち込む。client-IP を key にする limiter は **distinct key の無制限増加（CWE-770）** が DoS ベクタになる（many-source / spoofed-QUIC flood で OOM）——これは外部ライブラリ任せでは塞げない。

> 本 ADR は 2026-06-30 に proposed として起票し、2026-07-01 に実装確定して accepted とした。以下の設定キー名・manifest 形状・counter 名・status code marker・hardening は**実装済みの確定値**である（起票時の projected を本版で固定）。性能上の主張（hot path コスト等）は実測前のため数値は載せない。

## Decision

- **native の per-route（and/or per-IP）token-bucket rate limiter を fast path に導入する。** `[route.rate_limit] rate = N, burst = B` が、その route（または client-IP）へ向かうリクエストの token bucket を定義する。`proxy_core_inner` は **route 一致の直後・filter chain の前**でこの bucket を consult し、token が無ければ filter chain も forward も走らせず即 fast-fail する（front-door floor: flood を WASM CPU を一切使う前に弾く）。token-bucket の math は **host-native で完結**し、WASM 境界を一切跨がない。

- **token-bucket math は `apply_bucket` を再利用する（新規 external dependency ゼロ）。** [[000005]] / [[000026]] の host-native token bucket `apply_bucket`（純粋関数・I/O 非依存・`now_ms` に対し決定的・unit-tested）を pub 化し、native limiter（`plecto-control`）と per-filter host-ratelimit（`plecto-host`）で**同一実装を共有**する。`control → host` 依存は既存なので新規結合は無く、`apply_bucket` は WASM も filter-state KV も触れない純粋演算なので「native limiter は WASM 境界を跨がない」を満たす。`governor`（GCRA）採用は keyed map の無制限性を別途 wrap する必要がある上に新規 dep subtree を持ち込むため却下し、既に持っていて検証済みの math を再利用した。設定の `rate`/`burst` は `apply_bucket` の `Bucket { capacity = burst, refill_tokens = rate, refill_interval_ms = 1000 }` に写す。

- **設定語彙は `rate` + `burst`。** native の「粗い床」には Envoy local rate limit と同じ二ノブ（`rate` req/sec ＋ `burst` capacity）の friendly な surface を採る。per-filter host-ratelimit の `capacity` / `refill_tokens` / `refill_interval_ms`（[[000026]]）とは**あえて別語彙**にし、「native の床」と「policy 形状の filter limiter」を operator が混同しないようにする。`key` 既定は `route`、`[route.rate_limit]` 不在は無制限。

- **超過は 429 Too Many Requests で fast-fail ＋ `Retry-After`。** 上限到達時は `x-plecto-fault: rate-limited`（body `rate limit exceeded`）に加え `Retry-After`（`apply_bucket` が返す `retry_after_ms` の ceil 秒、advisory）を載せて即返す。**status は 429 を採用**し、[[000028]] の circuit breaker が返す **503**（`circuit-open` = upstream 飽和）とは**別の status / 別の fault 値**にする——「クライアントがレート超過（429）」と「upstream が過負荷（503）」を運用が `x-plecto-fault` と metrics で切り分けられる。`Retry-After`（起票時の再検討条件 (c)）は data が `apply_bucket` から無償で得られるため v1 で採用した。

- **per-IP のメモリ境界は fixed-size sharded テーブルで構造的に保証する（CWE-770）。** `key = client-ip` は peer アドレスを**固定 slot 数のテーブル**（`IP_SLOTS = 1<<16`、~1.5 MiB/route）に hash して数える。distinct IP ごとに entry を確保する map ではないので、many-source / spoofed-QUIC flood でも**メモリは O(1) で構造的に上限**され、攻撃者はメモリを増やせない（no-OOM テネットを policy でなく構造で満たす）。slot 衝突は 2 IP が bucket を共有し flood 時に互いを bounded に over-throttle するだけ（OOM ではない）。テーブルの hash seed は instance 毎の `RandomState` なので、攻撃者が victim の slot に衝突する source IP を事前計算できない。

- **client IP は接続 peer を key にし、IP 粒度は v4 /32 + v6 /64。** key は kernel が決める接続 peer（`peer.ip()`）であり、forgeable な `X-Forwarded-For`（[[000018]]）は使わない（edge モデル前提）。IPv4-mapped IPv6（`::ffff:a.b.c.d`、dual-stack listener）は v4 に collapse し `set_forwarded` が emit する XFF 値と一致させる。v6 は **/64** に丸め、1 host が自分の /64 内で address を回して bucket を回避するのを防ぐ。

- **設定値は build 時に fail-closed 検証する。** `rate == 0`（永遠に refill しない）/ `burst == 0`（1 token も持てない）は config typo として `build_active` が `ControlError::InvalidRouteRateLimit` で reject する（[[000026]] / [[000027]] の per-filter 検証と同列、CWE-20）。`refill_interval_ms` は 1000 固定なので zero-interval の経路は構造的に無い。

- **single-node-first を守る。** native limiter は**ノードローカル**な token bucket（このノードを通るリクエストのみカウント）。ノード横断の global / distributed rate limiting（外部 RLS）は、**保留中の foca/openraft 分散レイヤ（[[000008]]）の opt-in 後続**に置き、本 ADR では**引き込まない**。

検討した代替案と却下理由:
- **filter-only のまま（現状維持）** — 本 ADR が塞ぐ穴そのもの。filterless route に native ベースラインが無く、最低限のレート上限ですら**すべて WASM フィルタを要求**する。却下。
- **`governor`（GCRA）crate を採用** — keyed limiter を即座に得られるが、(1) keyed map を自動で bound せず CWE-770 を別途 wrap する必要があり、(2) `dashmap → hashbrown/lock_api/parking_lot_core/crossbeam-utils` の新規 dep subtree を deny.toml に持ち込み、(3) 既に持つ `apply_bucket` と二重実装になる。供給チェーンを小さく保つテネットに照らし、検証済みの自前 math 再利用を採った。却下。
- **`capacity` / `refill_tokens` / `refill_interval_ms` 語彙（host-ratelimit と統一）** — operator の token-bucket メンタルモデルを 1 本化できるが、native の「粗い床」には 3 ノブは重く、2 機構の区別を曖昧にする。friendly な `rate`/`burst` を採った。却下。
- **per-IP を bounded LRU で持つ** — cap までは正確な per-IP だが、持続的 many-IP flood で working set > cap になると churn eviction し各攻撃 IP が fresh burst で再入する（per-IP 制限を実質回避）上に LRU 機構（dep か hand-roll）が要る。fixed-size sharded テーブルは**攻撃者がメモリも回避余地も増やせない**構造保証で、衝突は bounded な over-throttle に閉じるため、こちらを採った。却下。
- **distributed RLS を今入れる** — single-node-first テネット（[[000008]]）に反する。分散は opt-in 後続。却下（再検討条件に置く）。
- **fixed-window counter** — window 境界で burst が 2 倍通る（境界跨ぎ問題）。token bucket は burst を `burst` で明示吸収でき平滑な refill を表現でき、Envoy local rate limit にも倣う。却下。

本決定は Fork 6（横断データプレーン関心事は native、user-policy は WASM）を inbound レート方向に具体化し、[[000029]] の Stage C（native rate limiting = global counters）を実体化する。**fail-closed**（bucket 枯渇は素通しせず 429）・**data-plane no-panic**（`apply_bucket` は saturating 演算、slot index は power-of-two mask で常に in-bounds、parking_lot の非 poisoning lock）・**single-node-first**・**operational simplicity** の各テネットに整合する。[[000027]] の inbound リソース上限を**レート軸で補完**し、[[000026]] / [[000028]] のいずれも supersede しない。

## Consequences

### Pros
- **filterless route が native の DoS ベースラインを得る**——WASM フィルタを書かずとも、route / client-IP 単位の素朴なレート上限を manifest だけで掛けられる。native proxy の Tier-0 レート防御の充足。
- **native（粗粒度）と filter（細粒度 policy）が相補**し、役割が床 vs 政策的天井で明確に分かれる。語彙（`rate`/`burst` vs `capacity`/`refill_tokens`/`refill_interval_ms`）も別なので混同しにくい。
- token-bucket の計算が **host-native で完結**し WASM 境界を跨がない（filter chain を起動しない素の front-door 経路）。`apply_bucket` 再利用で**新規 external dependency ゼロ**、math は既に unit-tested。
- **per-IP のメモリが構造的に bound**（fixed-size table、O(1)）。many-source / spoofed-QUIC flood でも OOM しない（CWE-770 を policy でなく構造で封じる）。hash seed が instance 毎なので targeted collision も不可。
- **429（rate-limited）＋ `Retry-After`** が 503（circuit-open）と別 status / 別 fault 値なので、「クライアント側のレート超過」と「upstream 側の過負荷」を運用が切り分けられ、legit client は back-off できる。

### Cons / Tradeoffs
- **ノードローカル**——水平展開時のノード横断 global レートは保証しない。N ノードに分散すると実効レートは概ね N 倍になり得る（distributed RLS は opt-in 後続、[[000008]]）。
- **rate limit 機構が 2 つ**になる（native の per-route / per-IP と、filter の host-ratelimit）。operator は「粗粒度の床は native、policy 的な細粒度は filter」という使い分けを理解する必要がある（ドキュメント負債。`crates/server/CONTEXT.md` に用語を分離して記録済み）。
- **per-IP の hash 衝突は bounded な over-throttle を生む**——2 IP が同 slot を共有すると flood 時に無関係な数ユーザを巻き込む（OOM ではなく上限内の誤巻き込み）。fixed-size の代償。
- v1 の key は route / client-IP のみ。より多次元のキー（header 値・path prefix・組み合わせ）は policy 寄りで filter（host-ratelimit）側か後続の native 拡張に委ねる。
- **再検討条件**:
  - (a) ノード横断の **global / distributed rate limiting** が要件化したら、外部 RLS もしくは分散レイヤ（foca/openraft, [[000008]]）の opt-in として設計する。
  - (b) route / client-IP 以外の**キー次元**（per-header・per-path-prefix 等）が native ベースラインに必要になったら拡張を検討（policy 寄りなら filter 側に寄せる）。
  - (c) ~~429 応答に `Retry-After` を載せるか~~ → v1 で採用済み。bucket 残量の可視化（`x-ratelimit-*` 系）は未対応で、必要なら後続。
  - (d) **per-IP は per-source の床であり aggregate load を縛らない**——`key` は route XOR client-ip の排他なので、多数 source からの総量も縛りたい場合は現状 `key = route` を使う。route 床 + per-IP fairness を 1 route で**重ねる（layering）**のは後続（fixed-table が per-IP の OOM は既に封じているため、layering は安全要件でなく機能要件）。
  - (e) **reload は limiter を作り直し node-local bucket を reset する**——operator-triggered（SIGHUP）かつ content-hash idempotent なので攻撃面ではないが、頻繁な reload は床を瞬間的に広げ得る。health-preserving reconcile（[[000017]]）と同様の「変わらない route の bucket 状態を跨いで保持」が要れば再検討。
  - (f) **per-IP keying は edge デプロイ前提**——信頼できる LB の背後では `peer.ip()` が LB のアドレスになり per-IP が collapse する。trusted-proxy append モード（[[000018]] / [[000022]] の後続 knob）が landing したら、信頼ホップ分だけ XFF から client を導出する経路と連動させる。
