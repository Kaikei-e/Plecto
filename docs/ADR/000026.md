# host-ratelimit の bucket 仕様を manifest（per-filter）へ移し try-acquire を (key, cost) に絞って host 強制にする

## Status
accepted

## Date
2026-06-27

## Affected Components
- filter contract（`wit/world.wit`）— `host-ratelimit` の `bucket` record を削除、`try-acquire` を `(key, cost) -> acquire` に slim 化
- extension plane（`plecto-host`）— per-filter `ratelimit_bucket` を load→instantiate で `HostState` に配線、`try-acquire` は host 設定 spec を適用、未設定は fail-closed deny
- control plane（`plecto-control`）— manifest `FilterEntry` に per-filter `ratelimit`、`load_options()` が `LoadOptions` へ
- example filter（`filter-hello`）— 新シグネチャに追従

## Context

`host-ratelimit`（[[000005]] 機構3）は token bucket の **refill / counting を host-native** に保ち（超ホット経路は WASM 境界を跨がない）、フィルタは「consult するか・どのキーで」を判断するだけ、という設計だった。しかし v0.1 の WIT は `try-acquire(key, cost, spec: bucket)` で、**bucket 仕様（capacity / refill）をフィルタが毎回渡していた**。

ここに穴がある——カウントが host-native でも、仕様がフィルタの**申告**である以上、**untrusted フィルタは巨大な `capacity` を渡して自分の limiter を実質無効化できる**。WIT 自身がこれを trust caveat として注記し（「once the manifest lands the spec moves host-side and `try-acquire` slims to (key, cost)」）、manifest 着地後の host 強制を予告していた。

その前提条件は満ちている——**宣言的 manifest は既に存在する**（[[000007]] / [[000008]] の `plecto-control`：`[trust]` / `[[filter]]` / `[chain]` の TOML）。よって本件は unblocked で、残っていたのは「spec を manifest へ移す」実装だけだった。これは [[000006]] の deny-by-default / fail-closed テネットを rate-limit に対して完成させる security 修正である。

## Decision

- **WIT 契約を絞る。** `host-ratelimit` から `bucket` record を削除し、`try-acquire: func(key: string, cost: u64) -> acquire` にする。**フィルタは key と cost だけを決め、spec を渡せない（＝申告できない）。** untrusted フィルタが capacity を盛って自分の制限を緩める経路を**構造的に消す**。
- **spec は manifest に per-filter で置く。** `FilterEntry` に `ratelimit: Option<RateLimitConfig>`（inline table `ratelimit = { capacity, refill_tokens, refill_interval_ms }`）を追加。operator が所有する宣言なので、フィルタは触れない。`FilterEntry::load_options()` が host の `LoadOptions::with_ratelimit_bucket(capacity, refill_tokens, refill_interval_ms)` へ写す。
- **host は per-filter の spec を Store に持たせる。** `LoadOptions` → `LoadedInner` → `HostState.ratelimit_bucket: Option<Bucket>` と load→instantiate で配線。`HostState::try_acquire(key, cost)` はその field の spec を `KvBackend::try_acquire` に渡す。token-bucket の計算（`apply_bucket`、per-key `(tokens, last_refill)` state）は[[000005]] のまま**不変**——変えたのは「spec の出所」だけ。
- **bucket 未設定は fail-closed で deny。** `host-ratelimit` を呼ぶフィルタに host 側 bucket が設定されていなければ `try-acquire` は `allowed=false` を返す。**untrusted フィルタが「host 制限が無いから素通し」を得ることは無い**（[[000006]] fail-closed）。
- **per-filter の名前空間は据え置き。** rate-limit state は引き続き `{filter_id}` 名前空間（[[000011]]）に載り、reload を跨いで生存する（同一 id は KV を継ぐ）。

検討した代替案と却下理由:
- **filter 申告のまま** — 本 ADR が塞ぐ穴そのもの。untrusted には効かない。却下。
- **manifest に named buckets ＋ `try-acquire(bucket-name, key, cost)`** — 複数 bucket を共有・参照できるが、WIT に `bucket-name` 引数が増え契約が太る。per-filter 1 bucket で現要件は足り、untrusted の override 不能性も同じく満たすので、小さい契約を採った（再検討条件 (a)）。
- **global default bucket を 1 つ** — 最小だが per-filter の差別化ができない。却下。
- **未設定で allow（no-op）** — fail-open で [[000006]] テネットに反する。deny を採用。

本決定は [[000005]]（Fork 6 の host-native rate limit）を**強制力まで完成**させ、[[000006]] の deny-by-default / fail-closed を rate-limit 経路へ具体化する。データプレーン no-panic は維持（`try-acquire` は untrusted 入力で panic せず、backend エラーも deny に写す）。

## Consequences

### Pros
- **untrusted フィルタが自分の rate-limit を無効化できなくなった**——spec が WIT から消え、host（operator の manifest）が唯一の出所。これが本 ADR の主目的。
- 契約が**小さくなった**（`bucket` record 削除、引数 3→2）。フィルタ側のコードも単純化（`filter-hello` は `try_acquire("default", 1)`）。
- 設定が**宣言的 manifest に集約**され（[[000007]] / [[000008]]）、監査可能・reload でアトミックに差し替え可能。
- token-bucket の中核（[[000005]] `apply_bucket`）は不変なので、性能・state 形式・per-filter 名前空間（[[000011]]）に影響なし。

### Cons / Tradeoffs
- **WIT 契約の破壊的変更**（`try-acquire` の引数）——全フィルタの追従と conformance 更新が要る（in-tree は `filter-hello` のみ、追従済み）。
- **bucket 未設定 = deny** は、`host-ratelimit` を import するのに manifest で bucket を書き忘れたフィルタの当該リクエストを全部 429 にする。fail-closed として正しいが、運用上は「import したら manifest に bucket を書く」規律を要求する。
- per-filter 1 bucket なので、複数の独立した limiter を 1 フィルタが持つことは現契約では表せない。
- **再検討条件**:
  - (a) フィルタが**複数 bucket を共有/参照**したくなったら named buckets（WIT に bucket 参照を足す）へ拡張する。
  - (b) 「未設定で deny」が運用上きつければ、**load 時に「`host-ratelimit` を import するフィルタは bucket 必須」を検証**する形（component の import 内省で fail-closed at load）へ寄せる。
